Multi Catch Block Exception

  • As the part of 1.7 version the following two concepts introduced in Exception Handling .
    • Multi-catch block
    • try with resources
Multi Catch Block:
  • Even though multiple Exceptions having same handling code we have to write a separate catch block for every Exception type in 1.6 version :
  • The problem in this approach are
    • 1. length of the code will be increased .
    • 2. Readability of the code will be reduced .
  • To resolve these problems SUN people introduced multi-catch block 1.7 version .
  • According to this we can write a Single catch block which can handle multiple different type of Exceptions simultaneously .
    Such type of catch block is called Multi-Catch block .
  • In multi-catch block there should not be any relation between Exception type(either child to parent or parent to child or same type)
    other wise we will get Compile time Error :
try with resources :
  • Until 1.6 version, whatever the resources we opened at the part of try block should be closed in finally block .
  • The problem in this approach are :
    • We should compulsory close the resources in finally block and hence complexity of the programming will be increased .
    • We should compulsory write finally block which increases the length of the code, so that readability will be reduced .
    • To overcome these problems SUN people introduced try with resources in 1.7 version .
    • The main advantage of try with resources, the resources which are opened as the part of try block will be will be closed automatically
      once the control reaches end of try block either normally or abnormally .
    • Hence we are not required to write finally block explicitly, which reduces complexity and length of the code .
Conclusion:
  • We can declare any no of resources, but these resources should be separated with ;(semicolon)
    try(R1;R2;R3){ ----- ----- ----- }
  • The resource reference variables are implicitly final . Hence within the try block we can not perform reassignment for that reference variable .
  • The resources should be Autoclosable .
  • A resource is said to be Autoclosable if the corresponding class implements java.lang.Autoclosable interface .
  • Autoclosable interface introduced in 1.7 version and it contains only one method i.e close() method .
  • Until 1.6 version, try should be followed by either catch of finally but form 1.7 version onwards we can take only try with resources without catch or finally block .
  • The main advantage of try with resources is we are not required to close resources explicitly and hence we are not required to write finally block .
  • Hence finally block will be come dummy .
